## Mobile PDF download/share — SHIPPED
On phones, "Download PDF" used to open the PDF inline (no save/share) because mobile browsers ignore `Content-Disposition: attachment` and iOS Safari ignores `<a download>` on blob URLs. Fix: a shared helper **`src/lib/pdf/deliver.ts`** routes the PDF Blob through the **Web Share API** when `navigator.canShare({files})` is true (native iOS/Android share sheet → Save to Files, AirDrop, WhatsApp, Mail) and falls back to a real file download on desktop/unsupported. `AbortError` (user dismissed the sheet) is treated as success.
- `deliverPdf(blob, filename, {title})` — share-or-download a Blob. `deliverJobPdf(jobId)` — fetch the server-rendered checklist PDF (`/api/pdf/[jobId]`, `credentials:'include'`, reuses the server `Content-Disposition` filename, maps 401/403 to friendly errors) then delivers it.
- Call sites converted from `window.open`/`<a target=_blank>`/manual-anchor: `JobChecklistEditor` (submitted-banner + read-only bottom buttons, `sharing` spinner), admin `jobs/[id]` (header + overflow-menu), client `jobs/[id]` (server component → new client `src/components/job/JobPdfButton.tsx`), cargo report `downloadCargoReport` (`render.ts`), **DRI Production Report PDF + .docx** (`DriReportBuilder.tsx` — removed its local `download()`), and the **Personnel CSV export** (`personnel/page.tsx`). `ClientCargoWorkspace` surfaces failures via toast.
- Helper generalized to **`deliverFile(blob, filename, mimeType, {title})`** with `PDF_MIME`/`DOCX_MIME`/`CSV_MIME` constants (so `.docx`/CSV share with the correct type); `deliverPdf` now delegates to it. Every app-generated downloadable form/export (checklist PDF, cargo sensor PDF, DRI PDF+.docx, personnel CSV) now shares on mobile / downloads on desktop. **Not touched (different mechanism — open/view a signed URL or external link):** stored-document viewers (vessel docs, personal credentials, job ops doc), the invoice web link, photo previews. **Office has no PDF button (excluded).** `/api/pdf/[jobId]/route.ts` auth/render untouched. Must be called from a click handler (gesture) over HTTPS. **Needs real-device verification on iOS Safari + Android Chrome.**
